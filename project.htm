The Design Task

 Overall design requirements:

  1. You are to design a simple serial communication system as depicted in Fig1 below. The system uses only 16 characters for communication. The characters are 16 hex digits from 0 to F. A message - string of 64 digits - is stored in  the internal memory on FPGA and send to the receiver when a pushbutton is depressed. The received string is stored then in the receiver's memory.
  2. It should be possible to display the original message char by char on the 3 digit dot matrix display by pressing a pushbutton. The received string can be displayed char by char on the 3 digit dot matrix display by pressing another pushbutton.
  3. You have to provide option to see both sent and received characters side by side on the display to compare them visually. This is necessary e.g. in milestone 2 if you do not communicate with another board but only use resources of your board.
  4. It is up to you to design the details of user interface based on the matrix displays, switches and pushbuttons but remember it needs to be user friendly. You need to design decoders for 5x7 matrix LED displays.
  5. The message needs to be encoded with error correcting Hamming code (7,4) which means 4 data bits are concatenated with 3 parity bits and send as a 7bit encoded word. The received message needs to be decoded.
  6. The Hamming encoded message is to be send as Manchester2 coded signal over the communication channel. The receiver needs to first decode the Manchester2 code to Hamming encoded binary and finally decode to binary for storage and display. 
  7. Hamming coding is based on parity calculation for groups of bits. You can find plenty of information on the WEB or in textbooks. A good start would be http://en.wikipedia.org/wiki/Hamming_code.
  8. Manchester2 coding is very simple and was presented in lectures. The Man2 encoded signal should be in frequency range 10kHz to 100kHz.
  9. One of main difficulties in communication is receiving asynchronous signals. The Manchester encoded signal from one board needs to be received by another board. Your receiver needs to oversample the incoming signal and extract the Manchester pattern. This must be designed carefully since the clocks of Nexys2 boards are jittery and you cannot rely on stability of the clock.
  10. You work individually on your design and almost the whole design can be tested on the same board. You will need another board to test robustness of your receiver. You will need to demonstrate inter-board communication in the final demo. It is advisable you find a partner to work on that part already from the beginning of this project.

Stage 2: MILESTONE 1

   

NOTES:

1. Fill in the missing signals on the block diagram in Fig2, expand blocks which you develop in this stage and all others that need it e.g. user interface or top_controller.

2. Manchester2 transmitter receiver can be efficiently implemented using FSMs or shift register based circuitry. Up to you which way you go but pay attention to the design of handshaking for fetching a new character to encode and to store a new decoded character. Remember oversampling for the receiver.

3. Fig2 above shows fetching a char to be encoded directly from memory. Remember that in the final design this will be coming from the Hamming encoder. Design the fetching control circuitry such that you can reuse it in your Hamming design later. The best solution is probably to make a dummy H encoder module just fetching from memory and passing a char to its output register.

4. You can develop and perform all the initial tests of Man2 on the same FPGA running both modules on the same clock to simplify the development. You oversample when receiving but your clock is the same as in the transmitter. If that works reliably you then move to tests (and possibly modifications) on two Neys2 boards.

5. It is a good idea to check clock jitter on the Nexys2 boards which you are going to use. You can do it in the lab. Do some calculations on how high oversampling is needed to avoid problems with jitter. Your required communication speed is low enough to solve the problem reliably.
